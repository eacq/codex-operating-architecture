# File Organization

1. Inventory before changing anything. Read only names, extensions, sizes, and timestamps by default; never inspect file bodies, credentials, browser data, `.git`, `.codex`, or hidden private-state folders unless the user explicitly scopes them.
2. Classify by purpose first, then lifecycle: `00-inbox`, `10-active`, `20-reference`, `30-output`, `40-archive`, and `90-private-local`. Keep source, generated output, documentation, assets, and runtime state separate.
3. Use lowercase kebab-case folder names and descriptive files. Prefer `YYYY-MM-DD-topic` for dated records and preserve extensions. Never overwrite; add a collision suffix and produce a preview manifest.
4. A global architecture iteration must invoke `scripts/Invoke-IsolatedGlobalExperienceIteration.ps1 -Apply -Replace`: validate the organized sandbox, then create an exact off-root pre-iteration snapshot of every replaceable local file and hash before replacement. If a later step fails, restore that snapshot with the external runner, remove iteration-added files, verify hashes, refresh global interfaces, record the failure, and require owner repair plus a complete rerun. A failed rollback is a blocking critical error. Only a successful replacement, double active validation, cleanup, interface validation, and lifecycle writeback completes the iteration.
5. For three or more non-linear categories, call `codex-image-workflow`'s visual planner and its format-selection gate. Use a sanitized GPT-generated explanatory image only when it materially improves understanding; keep Mermaid for small reviewable text structures, reserve SVG for genuine editable vector structure, and select PNG/JPG or a verified WebP derivative for ordinary raster delivery. Do not put names, paths, credentials, personal identifiers, or file contents in image prompts.
6. Treat organization rules like workflows: review evidence for a category or naming rule, then choose retain, refine, add, merge, split, deprecate, or remove. Record the decision and its validation in the project lifecycle; do not create overlapping buckets merely because one folder is untidy.
7. The global default is `managed_roots: ["."]`: every non-backup, non-protected file beneath the project root is eligible for organization during an iteration. `.git`, `.codex`, credential files, and external backup roots remain excluded. `Restore-GitTrackedWorkspaceLayout.ps1` reverses both tracked and eligible untracked moves in the isolated copy and restores affected references, so validation and Git always see the canonical layout.
8. When structure or image semantics change, refresh the README, manifest, and diagram together. Edit a still-valid image, regenerate after topology changes, and delete obsolete images.
9. Deletion requires evidence. `Remove-UnnecessaryOrganizationArtifacts.ps1` may delete only currently untracked disposable temporary/cache files and empty non-protected directories. Re-run candidate discovery on the active tree after replacement and again after validation, because validation may regenerate caches; never propagate deletion from a stale manifest. Quarantine every file outside the project with its relative path and SHA-256 before each deletion. In global iteration, use the Git-backed untracked/ignored candidate scan plus candidate-parent empty-directory cleanup to avoid broad active-tree scans while preserving the allowlist and protected-path checks. Never recursively delete an unknown folder or a tracked, runtime, dependency, credential, backup, `.git`, or `.codex` path.
10. Rollback protects the exact active state, not only Git history. `New-PreIterationRollbackSnapshot.ps1` stores replaceable tracked and untracked files outside the project; its optimized path builds a Git tracked/untracked/ignored manifest, excludes private/runtime/dependency roots, copies the filtered tree with robocopy, then verifies every manifest SHA-256. `Restore-PreIterationRollbackSnapshot.ps1` restores missing or changed files, removes only extra non-protected files, preserves protected state, and verifies every restored SHA-256. Never claim rollback from a Git bundle alone.
11. Use `scripts/Invoke-ContinuousIterationDiagnosis.ps1 -Target file-organization` when failures are deliberately serving as module diagnostics. Supply an explicit safe repair script; after each failure the controller records owner evidence, applies that repair, and reruns the complete file-organization probe. Keep retrying safe repairs until the probe passes, but stop on a failed repair action or any rollback-critical condition.
12. First-party disposable work must be architecture-owned by default. Use `scripts/Resolve-CodexRunRoot.ps1` for temporary fixtures, candidate clones, publication workspaces, caches, and scratch files; its default roots are `$ARCHITECTURE_ROOT\.runtime\tmp`, `work`, and `cache`, all ignored by Git. Only use OS temp, user-home caches, or package-manager-controlled locations when the external tool requires it, and then record the exception plus cleanup or retention boundary.

Read `references/organization-contract.md` for the naming, privacy, and apply contract.
